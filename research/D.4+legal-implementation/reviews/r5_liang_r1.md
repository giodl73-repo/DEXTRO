# Review 5 — Leah Liang (Reproducibility / Model Statute Implementation Lens)
**Paper**: D.4 — Adopting Algorithmic Congressional Redistricting: Legal Pathways, Constitutional Constraints, and Model Legislation
**Round**: R1
**Score**: 3/4 (Accept with Minor Revisions)

## Summary

My review focuses on the reproducibility and implementability of the model statute — specifically whether the statutory text provides sufficient operational specificity to achieve the reproducibility and manipulation-resistance goals the paper articulates. Legal papers proposing model statutes are often evaluated on their doctrinal analysis; I assess the statute from an implementation standpoint, asking whether a bureaucrat, court, or algorithm developer could actually use this text to build, certify, and enforce an algorithmic redistricting system. The statute is generally well-designed but has several implementation gaps that, if left unaddressed, would create administrative ambiguity and litigation vulnerability.

## Strengths

The model statute's property-specification approach is the right architectural choice for a durable federal mandate. Rather than mandating METIS or any specific algorithm, Section 2(a)(1) specifies the properties that a certified algorithm must satisfy. This allows technological progress to operate within the statutory framework without requiring statutory amendment — exactly the approach that has made the Huntington-Hill apportionment statute durable across 80 years.

The public accessibility requirement for source code and data (Sections 2(b)(2) and 2(c)(3)) is appropriate and well-placed in the certification and production provisions respectively. Importantly, both code and data publication are required — an advance over many algorithmic governance proposals that require code transparency but allow data opacity.

The adjustment period design (60 days for comment, 30 days for Bureau review, written explanation for each accepted or rejected adjustment) is operationally specific and creates a documented record for judicial review. The default-to-preliminary rule (Section 2(d)(3)) correctly places the burden of inertia on the final map adoption side, not on challengers to the preliminary map.

The judicial review provision's 60-day decision deadline (Section 3(c)) is appropriate for redistricting context and consistent with the urgency that redistricting timelines impose on courts.

## Concerns

**The certification procedure for the algorithm lacks reproducibility standards.** Section 2(b) requires the Bureau to certify "a redistricting algorithm that satisfies the requirements of subsection (a)(1)" and publish its source code. But algorithm certification for reproducibility requires more than code publication. A reproducible redistricting algorithm requires specification of: (1) software dependencies and version pins (e.g., which version of METIS or its equivalent, which version of the supporting libraries); (2) hardware environment (the algorithm may produce different results on different hardware architectures due to floating-point arithmetic differences); (3) random seed policy (if the algorithm uses stochastic components, what seeds are used, and are they fixed or documented?); and (4) input data version specification (which census data vintage, which TIGER shapefile version). Without these specifications, two different parties running the "same" certified algorithm on the "same" data may produce different maps — which is directly contrary to the paper's claim (and Section 2(a)(1)(D)) that the algorithm "produces identical output for identical input data." The statute should either require the Bureau to certify a fully reproducible environment specification alongside the algorithm, or explicitly address how reproducibility is verified and enforced.

**The "identical output for identical input" requirement in Section 2(a)(1)(D) conflicts with how graph partitioning algorithms work.** METIS uses multilevel graph coarsening and refinement algorithms that are deterministic only if the graph's internal representation (specifically, vertex and edge ordering) is deterministic. In practice, computing environments produce different adjacency graph representations depending on how geographic data is processed, which can cause METIS to produce different outputs even from "identical" input data. The paper's reproducibility claim is accurate for a fixed, pre-built adjacency graph with fixed vertex ordering, but not necessarily for a fresh run from raw census data. The Bureau's certification procedure must address how the canonical adjacency graph for each state is constructed and stored to ensure that reproducibility holds end-to-end, from raw census data to final district map. The statute should specify that the Bureau publish not just the algorithm's source code but also the serialized adjacency graphs used as algorithm inputs, so that reproducibility can be verified against a fixed intermediate representation.

**The adjustment submission process lacks anti-manipulation safeguards.** Section 2(d)(1) requires proposed adjustments to be "documented by reference to objective geographic, municipal, or community-of-interest factors" and prohibits introduction of partisan data. This is the right structural requirement, but the implementation mechanism is underspecified. The paper acknowledges in Section 4.3 that "geographic and community-of-interest adjustments have historically served as proxies for partisan manipulation" — yet the model statute provides no technical mechanism to detect or prevent this. At minimum, the Bureau's regulations (referenced in Section 2(b)(3) but not drafted) should specify a counterfactual analysis requirement: for each proposed adjustment, the Bureau should publish the partisan impact of the proposed change (based on publicly available election data) even if partisan impact cannot be the basis for acceptance or rejection. This transparency would create a public record that third parties can use to challenge adjustments that appear partisan in effect even if not in stated rationale.

**The statute does not specify who has standing to propose adjustments in Section 2(d)(1).** The adjustment provision states "any person may submit to the Bureau proposed adjustments." This is broad — it includes political parties, incumbent legislators, partisan advocacy organizations, and anonymous individuals. The absence of any restriction on who may submit adjustments, combined with the absence of any aggregation procedure for conflicting adjustments, creates a practical problem: the Bureau may receive hundreds or thousands of adjustment proposals for a single state. The statute should specify a process for the Bureau to prioritize, aggregate, and adjudicate competing proposals, including a procedure for resolving conflicts between proposed adjustments that cannot be simultaneously adopted.

**The 30-day timeline in Section 2(c) for running the algorithm on all 50 states is feasible but should be documented as such.** The statute requires the Bureau to produce preliminary district maps for all 50 states within 30 days of the apportionment announcement. The technical paper confirms that the algorithm runs in hours, not days, for any single state — but the Bureau would need to build the operational infrastructure (computing environment, adjacency graph generation pipeline, output rendering pipeline) before the census cycle. The statute's 180-day certification period (Section 2(b)) presumably covers this infrastructure buildup, but the statute should explicitly require the Bureau to demonstrate the operational pipeline — by running the algorithm on prior-census data — during the certification period, so that the 30-day production timeline is validated before it is legally mandated for the first live census.

**The VRA mode is not addressed in the model statute at all.** This is the most significant implementation gap. The paper's VRA analysis (Section 5) devotes substantial attention to the VRA mode — its edge-weighting mechanism, its Section 2 compliance properties, its Shaw v. Reno analysis — but the model statute contains no provision authorizing, requiring, or regulating the use of the VRA mode. Section 2(a)(1)(B) prohibits use of "data on the partisan affiliation, voting history, or electoral preferences of any person or geographic unit" — but minority population data is demographic, not partisan, and is not covered by this prohibition. A Bureau of the Census that wanted to certify an algorithm with a VRA mode would have no statutory authority to do so under the current draft. The statute needs an explicit provision: (1) authorizing the Bureau to certify an algorithm with a VRA-compliance mode; (2) specifying the data the VRA mode may use (minority population from the census P.L. 94-171 data); (3) requiring the Bureau to document the VRA mode's parameters; and (4) specifying the conditions under which the VRA mode must, may, or shall not be used for a given state.

## Minor Issues

The statute's findings in Section 1(a)(6) state that "algorithmic redistricting satisfies the requirements of section 2 of the Voting Rights Act of 1965 ... and produces on average more majority-minority districts than legislatively enacted plans." Congressional findings are not judicially binding on their own, but they can create a record that courts reference in statutory interpretation. If Section 2 VRA compliance is a matter of empirical demonstration (how many majority-minority districts are produced), the finding should be conditioned on VRA mode application, not asserted unconditionally for the baseline algorithm.

The effective date provision (Section 3(e)) — "This Act shall take effect with respect to redistricting conducted following the first decennial census conducted after the date of enactment" — means the statute cannot be used for court-ordered remedies in the pre-census period. This is inconsistent with Section 4.3 of the paper, which identifies court-ordered remedial use as an immediate pathway. The statute should include a separate provision authorizing pre-census court-ordered use of a preliminarily certified algorithm.

## Recommendation

Accept with minor revisions. The model statute makes a genuine contribution as a ready-to-use legislative template. Required revisions: (1) add reproducibility environment specification requirements to Section 2(b); (2) address the determinism/reproducibility problem for end-to-end runs from raw data; (3) add counterfactual partisan-impact transparency requirement to adjustment process; (4) specify adjustment submission standing and conflict resolution; (5) add VRA mode authorization and regulation provisions; (6) add pre-census court-ordered use provision.
