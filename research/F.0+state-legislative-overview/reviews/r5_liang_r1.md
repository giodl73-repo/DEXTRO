# Review 5 — Christina Liang
**Paper**: F.0: Algorithmic State Legislative Redistricting — A Research Program
**Round**: R1
**Score**: 2/4

## Summary

F.0 introduces the F research track and previews results from six companion papers. My review focuses on reproducibility: whether the data and software described actually exist, whether the computational claims are realistic, and whether a researcher could replicate the advertised results.

## Strengths

The paper is admirably explicit about data requirements. Section 3.2 states that block-group adjacency data "are not downloaded by default" and must be built explicitly using build_unit_adjacency.py. The estimated storage (4 GB for 50-state block-group adjacency, 40 GB for block-level) is quantified. The prerequisite command is provided verbatim. This is more transparency than many algorithmic redistricting papers offer.

The CLI examples (Section 3) are runnable commands, not pseudocode. The --chamber, --resolution, and --resolution auto flags are described with enough specificity that a replicator could verify whether the described flags exist in the binary.

## Concerns

**C1 — Block-group adjacency data availability unverified.** The paper states that "block-group adjacency data for the states studied in F.1--F.3 are available at the project repository." This claim is made in the conclusion (Section 7) but I can find no evidence in the paper, companion papers, or repository documentation that these files have actually been generated and deposited. The build_unit_adjacency.py script is described as generating them from TIGER/Line data, but the TIGER/Line block-group shapefiles themselves require downloading ~40 GB of source data (all 50 states' TIGER shapefiles at block-group resolution). A researcher attempting to replicate must run the download orchestrator, build the adjacency files, and then run the pipeline — a multi-hour or multi-day process. The paper should clearly state the end-to-end compute requirements for replication, not just the adjacency file size.

**C2 — build_unit_adjacency.py is not part of the production pipeline.** The CLAUDE.md for this project describes the pipeline as exclusively Rust-based after the 2026-04-29 cutover. The Python script build_unit_adjacency.py is listed as a preprocessing step (retained from the pre-cutover architecture), but its maintenance status is unclear. If this script is the critical preprocessing gate for all F-track results, its reliability needs to be explicitly documented. Is it tested? Does it handle all 50 states or only states tested during development?

**C3 — "Available at the project repository" is vague.** The conclusion's data availability statement ("All results in the F track are generated by the redist Rust binary (version tagged with the paper's release)") is technically non-committal: "version tagged with the paper's release" implies that a release tag will exist, but at draft stage this tag does not yet exist. A reviewer cannot verify the computational results against a non-existent release.

**C4 — Vermont and NH results reliability.** F.0 states that for NH (k/n_bg = 0.57) "block-group redistricting is feasible and produces valid non-empty districts." This is stated as a result but the mechanism — "population-weighted sub-unit assignment when a block group is the only unit available" — is described without citation to where this sub-unit splitting procedure is implemented or tested. Sub-unit splitting at block-group resolution requires block-level population data, which in turn requires that block-level data has been downloaded and processed. The cascade of data dependencies is not fully documented.

**C5 — Single-seed results across all 50 states.** F.0 previews that F.1's 50-state results use seed 42. For a research program claiming that algorithmic maps are comparable to enacted maps, single-seed results are insufficient. The paper should preview whether multi-seed sensitivity analysis is planned or whether the single-seed results are being presented as definitive.

## Recommendation

Major revision required. The reproducibility infrastructure described exists but is partially speculative (block-group adjacency files not confirmed deposited, sub-unit splitting procedure not documented, release tag not yet created). These are not fatal flaws but require explicit acknowledgment of what is and is not currently replicable.
