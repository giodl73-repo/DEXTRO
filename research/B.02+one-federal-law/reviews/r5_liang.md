## Summary

This advocacy paper proposes the Districting Integrity Act mandating the ApportionRegions redistricting algorithm, with the SHA-256 content-derived seed formula as the reproducibility mechanism. From a reproducibility and systems perspective, the paper's design for the computational component is thoughtful — the seed formula, convergence sweep, plan manifest, and hash chain together constitute a coherent auditable system. However, the paper makes several implementation claims about B.11 (ApportionRegions) that depend on a paper explicitly described as unimplemented, which creates a reproducibility gap at the heart of the proposal.

## Strengths

- **The SHA-256 seed formula design is cryptographically sound and constitutionally motivated.** The paper correctly identifies four properties of the formula: determined by census (not political actors), version-locked in statute, changes automatically across census cycles, and computationally preimage-resistant (unknowable before census release). The domain separation via "DIA_SEED_V1" version string is standard cryptographic hygiene and the paper explains why it is necessary (preventing collision with other SHA-256 uses of the same input). This is well-designed.
- **The integration of the convergence sweep into the statutory text (§2a(c)(2)) resolves a real implementation ambiguity.** Rather than specifying a fixed seed and hoping it's optimal, the statute specifies a starting seed and a convergence criterion (600 consecutive non-improving seeds), making the algorithm's termination condition part of the law. This is a technically sound design that closes the local-optimum concern without sacrificing reproducibility.
- **The plan manifest as a compliance artifact is correctly scoped.** The paper specifies that the manifest records "every parameter, the seed, the TIGER graph version, and the resulting district assignments, in a format that permits independent verification of compliance." Tying the EAC verification process to the manifest rather than to the map geometry is the right choice: it makes compliance a property of the process (auditable from the manifest) rather than a property of the output (which would require geometric verification).

## Weaknesses / P1 Items (Required Fixes)

- **The AR zero-variance claim is cited to B.11, which is listed as "planned, not implemented" in B.0.** The paper states (footnote, Section 3.3): "B.11, Table 3: seat-count variance across 50 seeds per state. Maximum observed variance: 0.00 in 47 states." If B.11 is a planned paper, Table 3 does not yet exist. A paper that is intended to form the basis of legislation must not cite unpublished or unrealised empirical results. This must be corrected: either (a) confirm that B.11 is complete and its Table 3 is an actual result with the binary and data available, or (b) restate the zero-variance claim as a theoretical prediction of the HH bisection tree structure, pending B.11's completion. If (b), the constitutional argument built on this claim must be softened accordingly.
- **The "30-minute" implementation claim for the 50-state run is not sourced.** Section 5.1 states the AR algorithm "runs a 50-state apportionment in approximately 30 minutes on a laptop." This figure may be correct for GeoSection (which has been run), but AR's k-way METIS calls (for non-binary prime factors like k=13 or k=19) have not been benchmarked because B.11 is not yet implemented. More specifically: a 19-way METIS partition of Texas's 5,000-tract subgraph is a substantially different computational problem from a sequence of binary bisections. The paper cannot claim a 30-minute runtime for an algorithm that has not been implemented and benchmarked. Runtime claims must be qualified as "estimated" or restricted to the GeoSection-based components that have been benchmarked.
- **The TIGER/Line adjacency graph is described as both an input and a fixed public artifact, but no version pinning is specified.** The statute text says "the federal TIGER/Line adjacency graph for the census year in which the apportionment was performed." But TIGER/Line shapefiles are updated annually within a census decade (the 2020 TIGER/Line Shapefiles were updated in 2021, 2022, and 2023 to correct county boundary errors and incorporate new census-designated places). If different states use different TIGER/Line vintage years for the same census decade, the "same" algorithm produces different adjacency graphs and potentially different maps. The statute must specify which vintage of TIGER/Line is canonical — either the initial redistricting data release vintage or a designated update — and the manifest must record the specific vintage used.

## P2 Items (Suggested Improvements)

- **The statutory text should include a `--convergence-threshold` enforcement provision.** Section 5 provides a statutory text specifying T=600, but it also notes that the reference implementation exposes a `--convergence-threshold` flag with the note that values below 600 "shall not be used to generate maps with legal effect." This enforcement mechanism is in the implementation documentation but not in the statutory text itself. Adding a clause to the statute that legal maps must be generated with `convergence-threshold ≥ 600` (or its equivalent) would prevent states from using a research-mode run (T=500) for a legal submission.
- **The plan verification Section (§2a(h)) should specify the manifest schema version.** The statute says maps must be submitted with "the plan manifest specified by the Election Assistance Commission." But the manifest schema (whatif-manifest v1) is a software artifact that will evolve. If the EAC promulgates regulations requiring whatif-manifest v1 but the redist binary ships v2 in 2031, compliance is unclear. The statute should either incorporate the manifest schema by reference to a specific version or delegate schema versioning explicitly to the EAC.

## Score: 3 / 4 — Minor Revision

The paper's computational design is thoughtful and the SHA-256 formula is the best available solution to the seed-manipulation concern. The B.11 citation problem (P1.1) is the most urgent issue: if the zero-variance claim is not empirically grounded, the constitutional argument for AR's uniqueness is correspondingly weaker. The runtime claim (P1.2) and TIGER/Line version pinning (P1.3) are both addressable with precise specification. On revision, this should be a strong accept for the legal audience it targets.
