**Status**: Proposed (R1 reviewed, P0+P1 fixes applied)

**New crate**: Not needed — extension to `redist-cli/src/bisection_runner.rs`

## Overview

Phase 1 CVD uses BFS hop-count as the distance metric. For coastal states (FL, WA, AK), elongated states (CA), or states with irregular geography, hop-count distance diverges significantly from geographic distance. A tract that is geographically close but connected via a long graph path will be incorrectly assigned in Phase 1.

Phase 2 adds `VoronoiMetric::Geographic`: Euclidean distance on projected coordinates for a geometrically correct Voronoi partition. The centroid update step uses the true population-weighted mean coordinate (not the medoid approximation required in Phase 1), making the algorithm both more accurate and simpler to implement.

**What changes from Phase 1**:

| Component | Phase 1 (GraphDistance) | Phase 2 (Geographic) |
|---|---|---|
| Distance metric | BFS hop count | Euclidean on Albers-projected coordinates |
| Seed update | Graph-distance medoid (approximate) | Population-weighted mean coordinate (exact) |
| New data | None | `tract_centroids: Vec<(f64, f64)>` in `LoadedGraph` |
| Seed prefix | `"CVD_INIT_"` | `"CVD_GEO_INIT_"` |
| CLI flag | `--cvd-metric graph-distance` | `--cvd-metric geographic` |

**Why not replace Phase 1**: Graph-distance CVD remains the default — it requires no additional data and works correctly for compact states. Geographic CVD is additive: it activates only when centroid data is present and `--cvd-metric geographic` is specified.

## 1. Centroid data

Tract centroids are the population-weighted centroid of the census tract polygon. They are derivable from TIGER/Line data and are stored in two ways:

**1. `LoadedGraph` field**:

```rust
pub struct LoadedGraph {
    // ... existing fields ...
    /// Population-weighted centroid of each tract polygon (WGS84 lon/lat).
    /// Empty if the centroid companion file is not present.
    pub tract_centroids: Vec<(f64, f64)>,
}
```

Loading: after loading the adjacency file, look for `{stem}_centroids.json` in the same directory. If present, load into `tract_centroids`. If absent, leave empty (Phase 1 fallback; no error at load time).

**2. Companion file** `{state}_centroids_{year}.json` in the same adjacency directory:

```json
{"centroids": [[lon0, lat0], [lon1, lat1], ...]}
```

One entry per tract in the same index order as the adjacency file. Index alignment is enforced: if `len(centroids) != len(adj)` the loader returns an error.

## 2. Projection

Continental US states use **EPSG:5070** (Albers Equal Area Conic, NAD83):
- Central meridian: −96°
- Standard parallels: 29.5°N, 45.5°N
- Preserves area; appropriate for population-weighted redistricting

| State | Projection | EPSG |
|---|---|---|
| Continental US (48 states + DC) | Albers Equal Area Conic, NAD83 | 5070 |
| Alaska | Alaska Albers Equal Area Conic | 3338 |
| Hawaii | Hawaii Albers Equal Area | 102007 |
| Puerto Rico | Puerto Rico and Virgin Islands | 32161 |

**Implementation** (simplified Albers, Reference: Snyder 1987 §14):

```rust
fn albers_project(lon: f64, lat: f64) -> (f64, f64) {
    // EPSG:5070 simplified Albers:
    let lon0 = -96.0f64.to_radians();  // central meridian
    let phi1 = 29.5f64.to_radians();   // standard parallel 1
    let phi2 = 45.5f64.to_radians();   // standard parallel 2
    let lon_r = lon.to_radians();
    let lat_r = lat.to_radians();
    let n = 0.5 * (phi1.sin() + phi2.sin());
    let c = phi1.cos().powi(2) + 2.0 * n * phi1.sin();
    let rho0 = c.sqrt() / n;
    let rho = (c - 2.0 * n * lat_r.sin()).max(0.0).sqrt() / n;
    let theta = n * (lon_r - lon0);
    let x = rho * theta.sin();
    let y = rho0 - rho * theta.cos();
    (x * 6_371_000.0, y * 6_371_000.0)  // scale to meters
}
```

State projection selection is automatic based on state FIPS code. Alaska (FIPS 02) uses EPSG:3338; Hawaii (FIPS 15) uses EPSG:102007; Puerto Rico (FIPS 72) uses EPSG:32161; all others use EPSG:5070. Projection constants must match those used in `redist-map` for consistency.

**Spherical approximation caveat**: The simplified Albers formula above uses a spherical Earth (radius 6,371,000 m). The NAD83/EPSG:5070 standard uses the GRS80 ellipsoid; the spherical approximation introduces ~0.3% maximum error for continental US coordinates. For redistricting at census-tract resolution, this error (≤3m for typical tracts) is negligible. An ellipsoidal implementation would use PROJ library bindings and is deferred to Phase 2.

## 3. Algorithm — Geographic Phase 2

```
split_subgraph_cvd_geographic(adj, pop, tract_indices, centroids, k=2, n_iter=20, base_seed):
  sorted = sorted(tract_indices)
  m = len(sorted)
  local_centroids = [centroids[i] for i in sorted]      // subset for this subgraph
  projected = [albers_project(lon, lat) for (lon, lat) in local_centroids]

  // k-farthest seed selection (Euclidean distance, no randomness consumed)
  cvd_seed = SHA-256("CVD_GEO_INIT_" || node_path.as_bytes() || "_" || base_seed:u64le) → u64le
  seed_0 = cvd_seed % m
  distances_from_0 = [euclidean(projected[seed_0], projected[i]) for i in 0..m]
  seeds = [projected[seed_0], projected[argmax(distances_from_0)]]

  for iter in 0..n_iter:
    // Voronoi assignment (Euclidean distance)
    for t in 0..m:
      assignment[t] = argmin_j(euclidean(projected[t], seeds[j]))

    // Post-hoc rebalance (same 200-iter boundary-swap as Phase 1)
    assignment = rebalance(assignment, adj, pop, balance_tolerance)

    // Centroid update (population-weighted mean of projected coordinates)
    for j in 0..k:
      tracts_j = [t : assignment[t] == j]
      total_pop = sum(pop[t] for t in tracts_j)
      new_seed = (
        sum(projected[t].x * pop[t] for t in tracts_j) / total_pop,
        sum(projected[t].y * pop[t] for t in tracts_j) / total_pop
      )
      // The centroid may not correspond to an actual tract.
      // Use nearest real tract in projected space as the updated seed.
      seeds[j] = projected[argmin_t(euclidean(projected[t], new_seed))]

    if seeds unchanged: break  // Note: the convergence check compares projected seed coordinates within an epsilon tolerance (ε = 1.0 meter) rather than exact equality, to avoid pathological non-convergence from discrete snap oscillations where the nearest-tract snap alternates between two equidistant candidate tracts.

  return (left_set, right_set)
```

**Key difference from Phase 1**: The centroid update computes the true population-weighted mean coordinate rather than the graph-distance medoid approximation. The result is a coordinate in projected space that may not correspond to a real tract; the nearest tract is used as the seed for the next iteration. This is exact Lloyd's algorithm on the projected point set.

**Contiguity**: Voronoi assignment does not guarantee contiguous districts for non-convex state geometries. The rebalance step detects and repairs disconnected components via BFS, consistent with Phase 1 behavior.

## 4. Seeding

New prefix to avoid collision with Phase 1:

```
cvd_geo_seed(base_seed, node_path) =
  SHA-256("CVD_GEO_INIT_" || node_path.as_bytes() || "_" || base_seed:u64le) → u64le
```

The prefix `"CVD_GEO_INIT_"` is distinct from Phase 1 `"CVD_INIT_"`. A test asserts both constants are present in source and are not equal. The k-farthest initialization (farthest-first traversal from a seeded starting tract) is used for Phase 2; this is deterministic given `cvd_seed` with no additional randomness.

## 5. Rust struct and API extension

**`VoronoiMetric` enum extension** (in `redist-cli/src/bisection_runner.rs`):

```rust
pub enum VoronoiMetric {
    /// Graph-distance (BFS hop count) — Phase 1, no extra data. Default.
    GraphDistance,
    /// Geographic Euclidean on Albers-projected coordinates — Phase 2.
    Geographic,
}
```

**`CentroidalVoronoiConfig`** gains no new fields; the metric variant already lives in `VoronoiMetric`.

**`run_centroidal_voronoi` signature** (unchanged from Phase 1 spec — `tract_centroids` was already `Option`):

```rust
pub fn run_centroidal_voronoi(
    adj: &[Vec<u32>],
    pop: &[i64],
    tract_centroids: Option<&[(f64, f64)]>,  // None → Phase 1 fallback
    k: u32,
    config: &CentroidalVoronoiConfig,
    base_seed: u64,
) -> (Vec<u32>, usize)
```

When `metric == VoronoiMetric::Geographic` and `tract_centroids.is_none()`, return `Err` immediately (caught at CLI layer before the call is made).

## 7. CLI

```bash
redist state --state NC --year 2020 \
  --structure centroidal-voronoi \
  --cvd-iters 20 \
  --cvd-metric geographic
```

```
ERROR: --cvd-metric geographic requires tract centroid data.
       Run: redist fetch --type centroids --states NC --year 2020
```

## 8. Audit chain

Every run appends to `runs/{label}/{year}/index.json`. Phase 2 adds three fields:

```json
"structure": "centroidal-voronoi",
"cvd_metric": "geographic",
"n_iter": 20,
"seed_init": "kmeans++",
"base_seed": 12345678,
"cvd_seed": 987654321,
"cvd_seed_formula": "SHA-256('CVD_GEO_INIT_' || node_path || '_' || base_seed:u64le)",
"node_path": "01",
"iterations_to_convergence": 11,
"convergence_warning": false,
"projection": "EPSG:5070",
"final_ec": 2847,
"final_pp_mean": 0.241
```

New fields relative to Phase 1: `"cvd_seed_formula"` (explicit formula string for auditor reproducibility), `"node_path"` (the bisection tree path string for this node — required for independent seed reproduction: `cvd_geo_seed = SHA-256('CVD_GEO_INIT_' || node_path || '_' || base_seed:u64le)`; without `node_path`, `cvd_seed` cannot be rederived from `base_seed` alone), and `"projection"` (EPSG code of the projection used; varies by state). `"cvd_metric"` was already present in Phase 1 output but defaulted to `"graph-distance"`; Phase 2 outputs `"geographic"`.

## 9. Test invariants

### L0 (inline unit tests)

- Same `base_seed`, geographic metric: identical assignment on two runs (determinism via Albers projection)
- `--cvd-metric geographic` without centroids returns `Err` with message containing `"centroid"`
- `albers_project(lon=-96.0, lat=37.5)`: `x.abs() < 1000.0` (within 1km of origin — central meridian should project to near-zero x in meters) and `y` is finite and non-negative (positive northing in the projection)
- Prefix constant `"CVD_GEO_INIT_"` in source is distinct from Phase 1 constant `"CVD_INIT_"` — hard-coded assertion
- Projected centroids are finite (no NaN/Inf) for all valid WGS84 coordinates in the continental US bounding box (lon ∈ [−125, −66], lat ∈ [24, 50])
- Index alignment: loading a centroid file with `len != len(adj)` returns `Err` mentioning `"length mismatch"`
- k-farthest seeding: `seed_1 = argmax(distances_from_0)` produces a different tract than `seed_0` for any connected subgraph with m ≥ 2

### L1 (integration, synthetic data)

- 4×4 grid with evenly-spaced synthetic centroids (1.0° apart): geographic CVD produces the same 2-way split as graph-distance CVD (no divergence for regular geometry)
- Elongated grid (4×1) with centroids: geographic CVD assigns leftmost 2 tracts to district 0, rightmost 2 to district 1 (identical to graph-distance on path graphs)
- Convergence: 100-iteration run gives the same result as 20-iteration run on regular grids (stable fixed point within budget)
- Contiguity: all k districts are contiguous for the 4×4 grid (rebalance enforces connectivity)

### L2 (`#[ignore]`, real data)

- NC 2020: geographic CVD mean Polsby-Popper > graph-distance CVD mean PP by ≥3% (geographic precision improves compactness for coastal NC tracts)
- FL 2020: geographic CVD mean PP > graph-distance CVD mean PP by ≥5% (FL panhandle tracts have large hop-count/geographic-distance divergence)
- `iterations_to_convergence < n_iter` for both NC and FL 2020 under default settings (algorithm converges within budget)

## 10. Open questions (deferred)

1. **Centroid source**: Should `tract_centroids` be fetched via `redist fetch --type centroids` as a new fetch type, or derived from existing TIGER/Line polygon data already in the data pipeline? (Latter avoids a new fetch step but requires polygon geometry at load time.)

2. **Automatic projection selection**: Should Alaska and Hawaii projection be selected automatically from state FIPS code, or should the user be able to override via `--cvd-projection epsg:3338`? The spec presumes automatic; confirm this matches `redist-map` behavior.

3. **International territories**: Can the projection logic extend to Puerto Rico (EPSG:32161) and Guam automatically? Guam lacks a well-known Albers; fallback to raw Euclidean may be acceptable.

4. **Centroid generation timing**: Centroid companion files (`{state}_centroids_{year}.json`) ARE generated during `redist fetch --type centroids` as a new fetch type. The fetch step computes population-weighted polygon centroids from TIGER/Line shapefile polygons and writes them to the adjacency directory alongside the existing `_adjacency_` and `_geoids_` files. This step is deferred to Phase 2 implementation.

5. **Interaction with bisection ensemble**: `BisectionEnsemble` over CVD subproblems varies `base_seed` per member. Verify that `cvd_geo_seed` (which depends on both `base_seed` and `node_path`) produces distinct seeds across ensemble members at each bisection node.
