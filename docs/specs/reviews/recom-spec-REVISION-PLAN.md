## Consolidated P1 Items

These items were raised by at least one reviewer as required changes before approval. Items raised by multiple reviewers are flagged with a count.

### A. Balance check / retry logic (raised by: Karypis, Duchin)

**Problem**: The spec's 50-random-draws-from-fixed-tree retry loop is not equivalent to GerryChain's ReCom implementation and produces a statistically biased sampler.

- Karypis: sampling 50 edges from the same tree is not helpful if no balanced cut exists in that tree; the retry budget should trigger a full tree resample, not another draw from the same tree.
- Duchin: GerryChain either enumerates all tree edges and picks uniformly among balanced cuts, or resamples the tree on failure. The spec's approach deforms the stationary distribution by over-sampling near-balanced edges.

**Required fix**: Replace the 50-retry inner loop with one of two specified strategies:
  1. Enumerate all tree edges, find all balanced cuts, pick uniformly among them (fails gracefully if zero balanced cuts exist → resample tree).
  2. Outer retry loop: resample the spanning tree up to N times per proposal; for each tree, enumerate all edges and accept the first balanced cut found.

Specify which variant is implemented and justify the choice relative to DeFord-Duchin-Solomon (2021).

---

### B. O(|E|) complexity claim for Wilson's algorithm (raised by: Karypis)

**Problem**: The pseudocode comment states `// Wilson's algorithm: O(|E|) expected time`. This is incorrect. Wilson's algorithm runs in O(cover time), which for planar graphs is O(|V| log |V|) expected, not O(|E|).

**Required fix**: Correct the comment to `// Wilson's algorithm: O(cover time) expected; O(|V| log |V|) expected for planar graphs (Lovász 1993)`. Remove or correct any prose claims of O(|E|) complexity.

---

### C. TX bipartition failure explanation (raised by: Duchin)

**Problem**: The spec implies that TX bipartition failures are caused by Python/GerryChain's implementation speed and would be resolved by a Rust port. This is incorrect — bipartition failures are a combinatorial property of the graph and population distribution, not an implementation artifact.

**Required fix**: Remove the parenthetical "(bipart failures)" from the TX row as a performance qualifier and add a note explaining that (a) TX at k=38 has a higher proposal rejection rate due to the low probability of balanced cuts in large merge regions, and (b) the Rust implementation addresses this by running more steps per second (so the *effective* rejection rate impact is smaller), but does not eliminate bipartition failures.

---

### D. Compactness extremum interpretation (raised by: Rodden, Stephanopoulos)

**Problem**: The spec mentions in the motivation that METIS plans fall at the 0.1–0.7th percentile of the cut-fraction distribution but does not explain what this means, whether it generalises across states, or how it affects the G-series papers' claims.

**Required fix**: Add a section "Ensemble Findings Preview" that:
  - States that METIS plans are more compact than ~99.3–99.9% of ReCom-sampled plans on cut-fraction.
  - Explains that this is expected: METIS minimises edge cuts (= maximises compactness) by design; ReCom samples uniformly over the feasible space, which includes many less-compact plans.
  - Notes that this does not imply partisan bias on cut-fraction — it implies the METIS plan is a compactness optimum, not a typical sample.
  - Specifies that the G-series papers require a *separate* partisan-outcome ensemble analysis (D-seats percentile, efficiency gap) not provided by this crate.

---

### E. Audit vs. generation use-case separation (raised by: Rodden, Stephanopoulos)

**Problem**: The spec lists three use cases (audit enacted map, characterise feasible space, compute diagnostics) but does not separate them in the CLI or output format. Mixing enacted-plan audit with METIS-plan audit creates legal ambiguity.

**Required fix**: Add a `--compare-plan` flag to the CLI (optional; default: the plan generated by `redist build` for the same state/year/version) and a corresponding `comparison_plan` field in the JSON output. Document that the "audit enacted map" workflow requires `--compare-plan <path-to-enacted-plan.json>` and that the METIS plan audit is a distinct workflow for methodological validation.

---

### F. Seed specification and reproducibility protocol (raised by: Liang)

**Problem**: The spec does not specify how `SmallRng` seeds are set, whether seeds are recorded in output, or how parallel chains receive independent seeds.

**Required fix**:
  - Add `--seed <u64>` CLI flag (if omitted, draw from OS entropy and print to stderr).
  - Add `seeds: Vec<u64>` field to `EnsembleResult` JSON output (one seed per chain).
  - Specify seeding strategy for parallel chains: `chain_seed = master_seed.wrapping_add(chain_index * 6364136223846793005u64)` (a standard multiplicative spread) or equivalent. Document this in the spec.

---

### G. Performance benchmarking methodology (raised by: Liang, Karypis)

**Problem**: The performance table shows target numbers, not measured benchmarks. The 2,500x claim is unverifiable.

**Required fix**: Add a Phase 2 benchmark task: run both GerryChain and the Rust implementation on identical hardware for each state (same graph, same number of steps, same population tolerance), record steps/second, and update the performance table with measured values. Specify the benchmark hardware in the spec. The table heading should change from "Rust target" to "Rust measured" after Phase 2 is complete.

---

### H. R-hat applicability to discrete statistics (raised by: Duchin, Liang)

**Problem**: Standard Gelman-Rubin R-hat is not valid for discrete redistricting statistics (seat counts, district labels). The spec applies R-hat to `cut_fraction` (continuous — acceptable) but the diagnostic module from `redist-analysis` may be applied to other statistics.

**Required fix**: Restrict the R-hat computation to continuous statistics (cut-fraction, pop-deviation). For any discrete statistic (seat counts, district-level partisan outcomes), specify that rank-normalised R-hat (Vehtari et al. 2021) is used, or document that R-hat is not computed for discrete statistics and an alternative convergence check (e.g., TV distance between chain marginals) is used instead.

## P2 Items (Suggested — Not Required for Approval)

- **Moran's I diagnostic** (Rodden): Add a geographic-sorting diagnostic to the output to separate compactness neutrality from partisan neutrality.
- **Burn-in period** (Liang): Add `--burn-in <steps>` CLI flag (default 1000) and record `burn_in` in JSON output to prevent initial-plan bias in ensemble statistics.
- **Full Hamming autocorrelation in JSON** (Liang): Output all 20 lags in the JSON array, not a truncated `...`.
- **Incremental adjacent-pairs update** (Karypis): Maintain a cut-edge set rather than scanning all edges per step to achieve the TX performance target.
- **ReCom variant documentation** (Duchin): Specify whether this implements original ReCom or bipartite ReCom and cite the relevant paper.
- **Legal use guidance section** (Stephanopoulos): Add a one-paragraph statement of what the ensemble output does and does not support as a legal claim.
- **Power analysis for ESS** (Liang): Justify 10K steps as sufficient for p5/p95 estimation, or specify a minimum ESS-per-quantile requirement.
