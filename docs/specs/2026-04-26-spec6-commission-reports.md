# Spec 6: Commission Reports and Audit Trail

**Date**: 2026-04-26
**Status**: Draft
**Dependencies**: All specs 1–5
**Depended on by**: Nothing (terminal consumer)

---

## Problem

A redistricting commission submitting a plan to a court or legislature needs:

1. **Machine-verifiable audit trail** — anyone can verify the plan was generated by this exact binary with this exact data
2. **Formal compliance report** — structured document covering all legal requirements
3. **Expert witness package** — data formatted for academic/legal expert analysis
4. **Public-facing summary** — readable HTML for public comment periods

Today `redist` produces JSON files and PNGs. No formal report output exists.

---

## New subcommand: `redist report`

```bash
# Full commission report for a single plan
redist report --label wa_house_draft1 --year 2020 --version WA_Plans \
  --format html pdf json \
  --out reports/wa_house_draft1_report/

# Full suite report
redist report --suite wa_commission_v1 --year 2020 --version WA_Plans \
  --format html --out reports/wa_commission_v1/

# Audit-only (no charts, just chain-of-custody JSON)
redist report --label wa_house_draft1 --audit-only --out reports/audit.json
```

## Report sections

### 1. Executive Summary (1 page)
- Plan name, state, year, chamber
- Number of districts
- Date generated, binary version
- Pass/fail for each legal requirement
- Comparison vs enacted plan (if available)

### 2. Population Equality (from Spec 1 + 3)
- Ideal population
- Per-district table: district ID, population, deviation %, pass/fail
- Max deviation vs tolerance
- Histogram of deviations

### 3. Geographic Constraints (from Spec 3)
- Contiguity: pass/fail per district + map of any violations
- County splits: count, list, comparison vs enacted
- Municipal splits: count, list, comparison vs enacted
- Preservation scores

### 4. Partisan Fairness (from Spec 4)
- Efficiency gap (value + 95% CI + threshold assessment)
- Mean-median difference
- Partisan bias
- Statewide vote/seat table
- Competitive districts count
- Note on election data source and year

### 5. Minority Representation / VRA (from existing analysis)
- Majority-minority district count and percentage
- Per-district minority population breakdown
- VRA compliance statement

### 6. Compactness (from existing analysis)
- Per-district Polsby-Popper, Reock, CHR
- Comparison vs enacted, vs statewide average
- Map of compactness scores

### 7. Comparison vs Enacted Plan (from Spec 2)
- Side-by-side metrics table
- Jaccard similarity (geographic stability)
- Which districts changed substantially

### 8. Nesting Compliance (from Spec 5, if applicable)
- Senate-to-house mapping table
- Nesting validation result

### 9. Audit Trail (from Spec 1 PlanManifest)
- Full `PlanManifest` reproduced verbatim
- SHA-256 hashes of all input files
- Command line used to generate this plan
- Verification instructions: "To reproduce, run: `redist state ...`"

### 10. Maps
- Final district map (all districts)
- Compactness choropleth
- Partisan lean choropleth
- Minority % choropleth

## Output formats

### HTML
Self-contained single-file HTML (no external dependencies). Embeds all charts as SVG and all maps as base64 PNG. Can be opened in any browser, emailed, printed.

Generated with a Rust template engine (`tera` crate — Jinja2-compatible) and the existing `resvg` SVG renderer.

### PDF
Generated by converting HTML → PDF via `wkhtmltopdf` subprocess or (preferably) `printpdf` crate for pure Rust PDF generation. Pure Rust preferred to avoid system dependencies.

### JSON
Machine-readable structured report. Same data as HTML but in JSON. Primary format for expert witness analysis in R/Python.

```json
{
  "report_version": "1.0",
  "generated_at": "2026-04-26T14:23:00Z",
  "plan": { /* PlanManifest */ },
  "sections": {
    "population_equality": { /* ... */ },
    "geographic_constraints": { /* ... */ },
    "partisan_fairness": { /* ... */ },
    "vra_compliance": { /* ... */ },
    "compactness": { /* ... */ },
    "comparison": { /* PlanComparison or null */ },
    "nesting": { /* NestingValidation or null */ },
    "audit": { /* full manifest + verification command */ }
  }
}
```

## Interoperability: external tool formats

The report command also exports plan assignments in formats compatible with other redistricting tools, so external auditors can independently verify using their preferred tool.

### `redist export`

```bash
redist export --label wa_house_draft1 --year 2020 --version WA_Plans \
  --format geojson shapefile csv mggg davesredistricting

# Produces:
#   wa_house_draft1.geojson     ← district polygons, compatible with PlanScore, DRA
#   wa_house_draft1.shp         ← ESRI shapefile for Maptitude, ArcGIS, QGIS
#   wa_house_draft1_tracts.csv  ← GEOID,district for any tool reading tract assignments
#   wa_house_draft1_mggg.json   ← GerryChain / MGGG Districtr format
```

**DRA format**: GeoJSON FeatureCollection where each Feature is a district polygon with `district_id` attribute.

**MGGG / GerryChain format**: JSON with `assignment` dict mapping GEOID → district integer.

**Shapefile**: dissolved district polygons, projected to EPSG:4269 (NAD83), with district ID and population attributes.

### Importing external plans for analysis

```bash
# Import a plan from DRA, PlanScore, or any GeoJSON source
redist import --file wa_house_external.geojson \
  --state WA --year 2020 --label wa_house_external \
  --version WA_Plans

# After import, run full analysis suite
redist analyze --label wa_house_external --types all
redist report --label wa_house_external --format html
```

Import workflow:
1. Parse GeoJSON district polygons
2. Assign each census tract centroid to the district polygon containing it (point-in-polygon)
3. Write as `final_assignments.json` in standard format
4. Write `manifest.json` noting `source: "imported"` and `source_file: "wa_house_external.geojson"`

Once imported, the plan is treated identically to a generated plan for all analysis purposes.

### Calling external analysis methods

For organizations with their own split-scoring or constraint methods, `redist` supports an **external analyzer hook**:

```bash
redist analyze --label wa_house_draft1 --types external \
  --external-analyzer "python my_org_splits.py {assignments_json} {output_dir}"
```

The external analyzer receives:
- `{assignments_json}`: path to `final_assignments.json`
- `{output_dir}`: path to write its output JSON

The output JSON must conform to the standard analyzer schema (have `analyzer` and `districts` fields). The report will include it as an additional section.

This allows organizations to plug in their own proprietary split-scoring algorithms, community-of-interest tools, or court-mandated analysis methods without modifying the `redist` binary.

## Audit chain of custody

The audit section includes a complete verification command:

```
To reproduce this plan exactly:

1. Install: redist v0.1.0 (SHA-256: abc123...)
2. Verify input data:
   sha256sum outputs/V3/data/2020/adjacency/wa_adjacency_2020.adj.bin
   # Expected: def456...
   sha256sum data/2020/tiger/tracts/53/tl_2020_53_tract.shp
   # Expected: ghi789...
3. Run:
   redist state --state WA --year 2020 --version WA_Plans \
     --districts 98 --chamber house --label wa_house_draft1 \
     --balance-tolerance 5.0 --seed 42
4. Verify output:
   sha256sum outputs/WA_Plans/2020/plans/wa_house_draft1/final_assignments.json
   # Expected: jkl012...
```

This is machine-generated from `PlanManifest` — no human writes it.

## Implementation

### New crate: `redist-report`

```
redist/crates/redist-report/
  src/
    lib.rs
    report.rs        ← Report struct, section assembly
    html.rs          ← HTML template rendering (tera)
    pdf.rs           ← PDF generation
    export.rs        ← GeoJSON, shapefile, CSV, MGGG export
    import.rs        ← GeoJSON/shapefile import → assignments
    audit.rs         ← chain-of-custody, verification command generation
  templates/
    report.html.j2   ← Jinja2/tera HTML template
```

### Dependencies
```toml
tera = "1"           # template engine
printpdf = "0.7"     # pure Rust PDF
sha2 = "0.10"        # SHA-256 for file hashing
geojson = "0.24"     # GeoJSON parsing/generation
shapefile = "0.6"    # already a dependency in redist-data
```

## Board Review Amendments (2026-04-26)

**[COVENANT] CRITICAL — Binary provenance must be independently verifiable**
The manifest hashes input data but not the binary itself. A court-appointed special master needs to independently obtain and verify the exact binary used.
Fix: `PlanManifest` gains:
```json
{
  "binary_download_url": "https://github.com/giodl73-repo/REDIST/releases/tag/v0.1.0",
  "binary_release_sha256": "hash from GitHub release page",
  "binary_build_commit": "git SHA"
}
```
Release process must publish SHA-256 hashes for each platform binary on the GitHub Releases page. The manifest verification command in the report includes: `curl -L {binary_download_url}/redist.exe | sha256sum` → must match `binary_release_sha256`.

**[COVENANT] CONCERN — External analyzer hook breaks audit integrity**
`--external-analyzer` passes assignments to a mutable external script. This breaks the self-contained reproducibility guarantee. Every report using an external analyzer has a gap in chain of custody.
Fix: External analyzer invocations must also SHA-256 hash the script file and record it in the manifest:
```json
{"external_analyzers": [{"script": "my_splits.py", "sha256": "abc123...", "command": "python my_splits.py {args}"}]}
```
Reports with external analyzers include a visible disclaimer: "This report includes analysis from external tools not part of the redist binary. Independent reproducibility requires access to the external scripts listed in the manifest."

**[LEDGER] CRITICAL — GerryChain export schema must be pinned to a specific version**
GerryChain changed its partition JSON schema between v2 and v3 (field `assignment` vs `part`; dict vs list format). An incompatible export silently produces wrong analysis.
Fix: MGGG export is pinned to GerryChain v2.3+ schema explicitly. The export format documentation states the target version. Add a round-trip integration test: `gerrychain_import → assignments → gerrychain_import` produces identical partition. Document: "Compatible with GerryChain ≥2.3. Earlier versions use different field names; see docs/specs/RPLAN.md for migration."
